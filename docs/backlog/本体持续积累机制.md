# 本体持续积累机制

> 入池日期：2026-06-22 · 状态：暂缓（构思）· 来源：体检模块深审延伸
> 铁律：入池=产品代码零残留。本机制**不得**与体检 MVP 绑死开发，先沉淀方案。

## 1. 为什么有这条需求

体检模块深审时暴露的因果链：**体检价值密度 ∝ 本体库丰满度**。本体库当前只有商品部分测试内容，需持续输入与积累。空本体 = 体检规则无处挂（指标无定义、字段无语义、表间无 link）。但「单开一个建本体的任务」没人有动力持续手工录入——会沦为一次性填表、之后烂尾。

**破局判断**：本体积累不该是独立工程，而应**搭已有工作流的便车 + 形成正循环**，让积累发生在用户本来就在做的事里。

## 2. 设计构思：三条便车 + 一条正循环 + 成熟度治理

### 2.1 三条便车（积累入口）

| 便车 | 触发点 | 复用的已有能力 | 缺口 |
|---|---|---|---|
| **数据登记便车** | 每登记/接入一个聚合集 | `onto from-aggregation`（零 LLM 投影 object/property，`/api/ontologies/:oid/objects/from-aggregation`） | 缺「登记即提议补本体」的钩子 + 草稿待确认队列 |
| **分析便车** | 探索/AnaX/专题中用到指标口径、发现勾稽关系 | onto-extract（pi LLM 抽取）/ metric_definitions | 缺「一键提为本体条目」入口 + 从分析产物反哺 |
| **体检便车（正循环核心）** | 体检遇到「想查但本体没建模」 | 体检引擎 needs vs 数据能力匹配（V-HEALTH1） | 缺「本体/数据缺口」落库 + 体检报告「本体覆盖率 + 待补清单」一栏 |

### 2.2 正循环

```
本体丰满 → 体检规则可挂、覆盖率↑ → 体检价值↑
   ↑                                        │
   └── 用户按「待补清单」补本体 ← 体检产出本体缺口 backlog ←┘
```

体检不只是**消费**本体，还**驱动**本体积累：每次体检把「按现有规则想查、但缺 metric 定义 / 缺 property 语义 / 缺 link」记成缺口，聚成「本体补全 backlog」，既是体检覆盖率报告，也是本体积累的待办源。两个模块互相喂养，而非体检干等本体建好。

### 2.3 成熟度治理

复用 onto 已有 `status: draft→active→archived`。积累 = 把 draft 逐步提升为 active：
- from-aggregation / 缺口驱动产出的条目入 `draft`；
- 人确认 / 被分析实际使用 → 升 `active`；
- 体检覆盖率只计 `active` 本体，避免半成品虚高。

可选加「置信度 + 来源溯源」（沿用 onto confidence 字段 + nano `_calibrate_confidence` 思路），标注条目来自数据投影/文档抽取/人工/缺口补全。

## 3. 与体检模块的边界（关键）

- 体检 **MVP 不依赖本机制**：本体空也能跑数据质量类规则，同时产出缺口清单。本机制是「让体检越用越准」的增强，不是前置。
- 体检侧只需做**最小钩子**：缺口落库 + 报告里展示覆盖率/待补清单（已写进 V-HEALTH1 的 needs 匹配「记缺口」+ V-HEALTH3 报告，属体检卡范围，非本机制）。
- 本机制**真正的工程**（登记钩子 + 草稿确认队列 + 分析反哺入口 + 成熟度流转 UI）独立立项，待捞出。

## 4. 将来开发要点（捞出指引）

1. **先验缺口循环**：体检便车的「缺口→待补清单」是否真能驱动用户补本体（小步验证产品假设），再决定投入登记/分析两条便车。
2. **数据登记便车**：在聚合集登记/接入流程挂 `from-aggregation` 自动提议钩子 + 草稿待确认 UI（onto 已有 from-aggregation，补触发与队列）。
3. **分析便车**：探索/AnaX 产物加「提为本体条目」动作（metric 口径 / link 关系 → onto draft）。
4. **成熟度流转**：draft→active 的确认入口 + 覆盖率口径 + 可选置信度溯源。
5. **避免过早自动化**：先半自动（提议→人确认），积累出真实模式再考虑自动升级，勿一上来全自动污染本体。
