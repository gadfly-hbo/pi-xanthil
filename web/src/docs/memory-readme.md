# 记忆库模块 操作手册

> **模块定位**：「记忆库」是 pi-Xanthil 的长期上下文中枢，负责把跨会话沉淀下来的约束、经验、事实、指标口径、业务背景、调用轨迹和知识关系组织起来，并在后续日常对话、专题流水线和工作流运行时按需注入给 LLM。
>
> 📅 更新日期：2026-06-29
>
> 本文按二级 tab 的真实使用顺序讲解每个 tab 的功能、典型操作与样例。记忆库不是资料库：资料库保存外部文档；记忆库保存“后续分析必须记住的判断、口径、约束和经验”。

---

## 0. 总览

进入路径：顶部主导航选中「**记忆库**」（`activeTab === "rule_memory"`）。

二级 tab 按记忆治理链路排列（`RULE_MEMORY_SUB_TABS`）：

| # | 二级 tab | 一句话定位 | 与 LLM 关系 |
|---|---|---|---|
| 1 | **统一记忆** | 管理 `memory_items`：约束、经验、事件、事实投影、候选复核与 Skill 提案 | 启用后进入统一记忆注入 |
| 2 | **onto-knowhow** | 维护指标口径与参照标准文件 | 启用后进入指标/标准 prompt |
| 3 | **业务环境** | 维护组织、业务现状、术语、约束、历史与目标 | 启用后进入业务背景 prompt |
| 4 | **trace** | 回放调用轨迹、失败归因、记忆注入快照，并从 trace 提炼规则候选 | 部分提炼动作调用 LLM，候选需人审 |
| 5 | **知识图谱** | 把规则、指标、业务环境和报告关系可视化为节点/边 | 图谱可参与记忆注入；AI 语义提取会调用 LLM |
| 6 | **readme** | 本说明文档 | 无 |

顶部主导航右侧的「**记忆库 on/off**」控制本模块是否注入下一次 LLM 调用。关闭后，各 tab 仍可维护内容，但不会影响模型推理。

> ⚠️ 数据安全边界：记忆库内容应来自用户手写、LLM 衍生结论、聚合事实或报告产物；不要把 `draw_data` 原始行、用户级明细、样本值列表写入记忆。数据探索模块不会向记忆库写入原始样本。

## 1. 统一记忆（`rules`）

### 功能

- 管理统一记忆真源 `memory_items`，类型包括：
  - `constraint`：必须遵守的硬约束 / 红线 / 输出纪律。
  - `experience`：可复用经验、分析偏好、成功做法。
  - `episode`：一次性事件、阶段性背景、短期状态。
- 展示 `fact (投影)`：来自业务环境、指标口径、标准文件等结构化事实，只读展示。
- 处理 `review`：D-INGEST 候选复核队列，避免自动沉淀直接污染记忆。
- 处理 `提案`：Safe Distiller 生成的子技能提案，需人工确认。
- 支持新建、编辑、启用/停用、正负反馈、删除。
- 支持 tag 筛选，推荐 tag 前缀：
  - `task:` 任务类型
  - `industry:` 行业
  - `method:` 方法论
  - `data:` 数据主题
  - `problem:` 问题类型
- 支持「预览注入」：按 `chat / workflow` scope、tag 硬过滤和 query boost 查看最终会注入的 prompt。
- 支持 Dream Worker 维护：
  - 「立即维护」：纯算术升/降 confidence、老化退役，先预览再应用。
  - 「查看老化信号」：扫描卡冲突与 supersede 旧引用残留，零 LLM。
  - 「升级 Skill」：把高频 `experience` 簇升级为 Skill registry candidate，不自动启用。

### 典型操作

1. 点击「新建」写入一条 `constraint` 或 `experience`。
2. 填写标题、正文、scope 和 tags。
3. 保存后确认左侧勾选为启用状态。
4. 点击「预览注入」，选择 `chat` 或 `workflow`，用 tag/query 模拟检索结果。
5. 如果记忆过期或冲突，点击「立即维护」或「查看老化信号」先看预览，再决定是否应用。

### 案例：写入分析输出纪律

```
type: constraint
scope: chat
title: 报告中的数字必须带来源
body: 输出报告时，每个关键数字必须标注来源：工具结果、聚合表、报告引用或人工输入；不允许用“约”“大概”等词替代来源。
tags: task:report, method:evidence, problem:hallucination
```

保存并启用后，后续日常对话打开「记忆库 on」时，这条约束会被检索进系统上下文，提醒模型在报告中标注数字来源。

### 案例：把高频经验升级为 Skill 候选

多次分析都沉淀了 `method:rfm`、`task:segmentation` 的经验后：

1. 点「升级 Skill」。
2. 查看 eligible cluster 列表。
3. 确认后执行升级。
4. 新 Skill 只进入 registry candidate 状态。
5. 到「控制 / 实验场 / Skill」评测候选，通过后再激活。

这条链路不会自动启用新 Skill，避免未经评测的经验直接影响生产分析。

## 2. onto-knowhow（`indicators`）

### 功能

- 维护分析指标口径：
  - 名称
  - 分类
  - 含义
  - 公式
  - 口径
  - 单位
- 维护参照标准文件：
  - 标准文件名称
  - 绝对路径
  - 用途说明
- 支持本工作区启用/停用；全局池删除会影响所有工作区。
- 提供「注入 prompt 预览」和「复制 prompt」。
- 指标真源使用 `metric_definitions`，标准文件仍走 analysis standards；UI 统一呈现为指标体系。

### 典型操作

1. 点「指标」新增口径。
2. 填名称、分类、公式、口径和单位。
3. 保存后在列表中点「本工作区启用」。
4. 在预览区确认 prompt 是否包含该指标。
5. 需要把外部规范作为引用时，点「标准文件」登记绝对路径和用途说明。

### 案例：复购率口径

```
名称：30 日复购率
分类：会员运营
含义：首购后 30 天内再次下单的用户占比
公式：30 日内复购用户数 / 首购用户数
口径：按 user_id 去重；退款订单剔除；统计窗口从首购支付成功时间开始
单位：%
```

启用后，模型在解释复购分析时会优先使用这条口径，而不是自行猜测分母和时间窗口。

### 案例：参照标准文件

```
名称：会员等级定义
分类：会员体系
绝对路径：/Users/.../clean_data/member-tier-definition.md
用途说明：用于解释 V1/V2/V3 等级划分和权益差异
```

标准文件通常只注入路径与用途说明；需要读取正文时，由具体分析链路按权限读取，不要在记忆条目里复制整份敏感文档。

## 3. 业务环境（`business_context`）

### 功能

业务环境用于补齐模型不知道、但做判断必须知道的真实背景。当前分为六类：

| 类别 | 用途 |
|---|---|
| 组织/主体 | 公司、部门、行业、规模、商业模式 |
| 业务现状 | 当前阶段、近期变化、核心矛盾 |
| 术语/口径 | 内部黑话、特殊定义、业务解释 |
| 约束/红线 | 不能做什么、合规要求、敏感点 |
| 历史/背景 | 已知结论、历史事件、踩过的坑 |
| 目标/期望 | 这次分析或决策真正要解决什么 |

每条业务环境都可按工作区启用/停用。启用后会随统一记忆、指标体系一起进入记忆注入链路。

### 典型操作

1. 选择一个类别，点「添加」。
2. 用一句话标题概括事实。
3. 在正文写清楚背景、限制或目标。
4. 保存后确认「本工作区启用」。
5. 在「注入的 system prompt 片段」确认最终内容。

### 案例：组织与目标

```
类别：组织/主体
标题：品牌处于高端化转型期
内容：当前品牌正从低价促销转向高端会员运营，短期 GMV 下滑可以接受，但高净值会员留存不能恶化。
```

```
类别：目标/期望
标题：本季度优先保护高净值会员
内容：所有运营建议优先评估对高净值会员复购、客单价和投诉风险的影响；不要只按短期 GMV 排序。
```

启用后，后续策略建议会更倾向于解释“为什么不能简单降价促销”。

## 4. trace（`trace`）

### 功能

- 查看最近调用轨迹，包括 session、run、failure、event 等。
- 用 KPI 卡片观察近 14 天运行趋势。
- 查看最近事件流、Session / Flow Timeline。
- 聚合失败类型，辅助判断是工具失败、模型失败、路径失败还是上下文失败。
- 查看「记忆注入检查器」：
  - 最近 20 次注入快照
  - 每个来源是否被选中
  - token 估算
  - 忽略原因
- 查看「检索失败诊断」：
  - 记忆为什么没被召回
  - 哪个 source 出现问题
  - 对应 target 与失败原因
- 从 trace 提炼规则候选：
  - 点击「更新规则提炼」
  - 选择候选
  - 暂存
  - 提交审批
  - 批准后写入统一记忆

### 典型操作

1. 在搜索框输入 session、target 或错误关键词。
2. 用 event/status/error 下拉过滤事件。
3. 点击事件查看 timeline。
4. 如果发现模型反复犯同类错误，点击「更新规则提炼」。
5. 只选择证据充分的规则候选，提交审批后再批准。

### 案例：修复“忘记引用来源”

一次报告生成没有引用数据来源：

1. 在 trace 中搜索该 session。
2. 打开事件流，确认报告生成成功但输出缺少来源。
3. 在「规则提炼」里生成候选。
4. 选择“报告数字必须标注来源”的候选并提交审批。
5. 批准后进入统一记忆。
6. 下一轮同类报告会在记忆注入中看到这条约束。

### 注意

- trace 是诊断入口，不是批量编辑入口。
- 规则提炼可能调用 LLM，但候选不会自动写入；必须通过 guardrails 与人工批准。
- 不要把 trace 中的敏感样本复制进记忆正文。

## 5. 知识图谱（`knowledge_graph`）

### 功能

- 从规则、指标体系、业务环境和工作流报告中同步节点与关系。
- 图谱视图展示节点、边、聚类和关系。
- 列表视图用于查看节点明细。
- 可隐藏节点、查看详情、手工连边、删除所选边。
- 点击「更新图谱」从已登记的记忆资产中摄入结构化关系。
- 点击「AI 语义提取」可从报告中提取概念节点和关联边（每次最多处理 5 篇）。

### 典型操作

1. 先在统一记忆、onto-knowhow、业务环境中维护关键条目。
2. 切到「知识图谱」点「更新图谱」。
3. 在图谱中搜索关键词，例如 `复购`、`高净值会员`。
4. 点击节点查看详情。
5. 对明显无关或过时节点执行隐藏；必要时手工补边。

### 案例：让“复购率”连接到“高净值会员”

1. 在 onto-knowhow 新增「30 日复购率」。
2. 在业务环境新增「本季度优先保护高净值会员」。
3. 更新图谱。
4. 如果两者未自动连接，手工从「高净值会员」连到「30 日复购率」，关系描述为“核心监控指标”。

后续分析时，图谱能帮助模型理解指标和业务目标之间的关系，而不是把指标当成孤立定义。

## 6. 推荐工作流

### 新项目初始化

1. 在「业务环境」写入组织、业务现状、术语和目标。
2. 在「onto-knowhow」录入核心指标口径。
3. 在「统一记忆」写入强约束，例如数据安全、报告格式、引用纪律。
4. 打开「预览注入」确认 token 规模可控。
5. 顶栏打开「记忆库 on」。
6. 回到「日常」或「专题」开始分析。

### 一次分析结束后

1. 到 trace 查看失败和注入快照。
2. 对重复出现的问题提炼规则候选。
3. 把长期有效的候选批准进统一记忆。
4. 对只影响本次项目的内容写成 `episode`，并设置合适 scope。
5. 对高频经验定期尝试「升级 Skill」，但必须经过评测再激活。

### 定期治理

1. 在统一记忆点「查看老化信号」。
2. 处理卡冲突和旧事实引用。
3. 点「立即维护」预览 confidence 调整。
4. 只在预览合理时应用。
5. 清理不再适用的业务环境和指标口径。

## 7. 注入链路

记忆库的内容不会因为存在就自动进入每次 LLM 调用，必须同时满足：

1. 相关条目处于启用状态。
2. 顶栏「记忆库」开关为 on。
3. 当前任务的 scope、query、tag 与条目匹配。
4. token budget 允许放入。

简化链路：

```
统一记忆 memory_items
业务环境 business_context
指标/标准 onto-knowhow
知识图谱 knowledge_graph
        │
        ▼
memory-injection 检索与裁剪
        │
        ▼
system prompt 附加区
        │
        ▼
日常对话 / 专题流水线 / 工作流节点
```

你可以在「统一记忆 → 预览注入」和「trace → 记忆注入检查器」中核对实际发生了什么。

## 8. 安全与维护注意事项

- **不要存原始行**：记忆正文中不要保存 `draw_data` 原始行、用户级明细、样本值列表。
- **不要把临时判断写成永久规则**：短期背景用 `episode`，长期纪律才用 `constraint`。
- **不要一次启用太多宽泛记忆**：宽泛条目会挤占 token，也会干扰更具体的记忆。
- **优先写可验证事实**：指标口径、业务约束、成功经验要有清楚依据。
- **冲突先停用再合并**：发现两条记忆互相矛盾时，先停用低置信条目，再写 supersede 或合并后的新条目。
- **Skill 升级不等于启用**：从记忆升级出来的 Skill 只是 candidate，需要实验场评测和人工确认。
- **知识图谱不是新真源**：图谱是关系视图，真源仍在统一记忆、指标体系、业务环境和报告产物里。

## 9. 快速排障

| 现象 | 可能原因 | 处理方式 |
|---|---|---|
| 顶栏显示「记忆库 none」 | 没有可注入内容或全部未启用 | 到统一记忆 / 业务环境 / onto-knowhow 启用条目 |
| 记忆库 on 但模型没遵守 | 条目未被召回、token 被裁剪、规则写得太泛 | 看「trace → 记忆注入检查器」和「统一记忆 → 预览注入」 |
| 旧规则仍影响输出 | 旧条目还启用，或 supersede 后未退役 | 查看老化信号，停用旧条目或应用维护 |
| 指标口径不对 | onto-knowhow 中口径缺失或未启用 | 新增/编辑指标，确认本工作区启用 |
| 业务背景没被模型考虑 | 业务环境未启用或顶栏开关关闭 | 启用对应条目并打开「记忆库 on」 |
| 图谱为空 | 还没有可摄入节点，或未点击更新 | 先维护记忆/指标/业务环境，再点「更新图谱」 |
