# 动态技能注入 + 子技能蒸馏（瘦 context · Skill Engineering）

> 入池日期：2026-06-23 · 状态：暂缓（方案）· 来源：桌面文档《skill技能打磨与动态技能研究》「动态技能解锁机制」+ SkillsInjector（arxiv 2605.29794）+ 技能演化综述（arxiv 2606.11435：Trace2Skill/SkillX/SkillRouter）
> 铁律：入池=产品代码零残留。
> 说明：文档「动态技能解锁」未对到单篇腾讯论文（综述疑为 WorkBuddy/TokenHub 产品报道），故以主题最贴合的真实论文为落地依据。本条对 pi-xanthil 是**全新能力**（现有 Harness 专题未覆盖运行时技能注入）。

## 1. 为什么有这条需求

pi-xanthil 现状：工作流/对话偏「**全量规则注入**」——把记忆/规则尽量塞进 system prompt（胖 context）。文档与论文共同结论：**加越多 skill 反而越差**（注意力分散 + skill 竞争）。趋势是 **瘦 context + 每步动态检索注入 2-3 个 micro-skill**。SkillsInjector 实测：tau2-airline +3.9pp、SkillsBench +6.1pp、ALFWorld +7.3pp，且每任务省 6-8 条交互消息。

对 [[多 agent 总控分工]] 的 `MultiAgentExecutionPane`（E）和 [[需求池机制]] 里 RulesPane（D）是直接升级点（文档点名这两处）。

## 2. 三块机制（论文骨架 → pi-xanthil 对位）

### 2.1 运行时动态注入（SkillsInjector）— 核心
两组件协同把「技能注入」从静态检索改为**按任务/按步构建 context**：

| 组件 | 作用 | 关键点 |
|---|---|---|
| **context planner** | 按**执行效用** Δ(t,s)（相对无技能基线的增益）打分选技能，**非语义相似度** | 自适应预算：per-domain 阈值 `𝒞t*={s: f̃θ(t,s) ≥ τd*}`，每任务admit 数量可变（如 82 技能库只选 4 个）|
| **set-aware renderer** | 按**共注入邻居**改写每条技能描述（加 scope 子句、减重叠） | 瘦 context：不扩 token 反降描述重叠 |

- 执行效用 `𝒰(t,C̃)=𝔼[r(τ)]`；选择算子 `A(t,𝒮)→(𝒞t, ρt)`。
- 训练：alignment loss(KL) + pairwise preference loss；hard negative=语义近但 Δ 低的技能（逼模型区分「看着像但没用」）。
- **对 pi-xanthil 关键启发**：选技能要按「这条 skill 在本任务真带来增益吗」而非「描述像不像」——pi-xanthil 现有 skill-retrieval.ts 多半是语义检索，需补执行效用信号。

### 2.2 子技能蒸馏 + 去重/多样性（Trace2Skill / SkillX）
- **Trace2Skill**：从多条执行轨迹的成功/失败各生成 targeted patch，**合并冗余 fix 成单一无冲突 skill 文件**。
- **SkillX**：合并相似 skill / 拆解复杂 skill / 评估泛化性，建「skill 相似关系图」判 reuse 还是 merge。
- **SkillRouter 警示**：只用 skill 名+描述选路由，准确率比用 full body **掉 31-44%**——选技能必须看技能体，不能只看标题。
- 文档「2-5 动作子技能 + 去重多样性」对位：从成功轨迹切高频动作序列为 micro-skill，去重后**优先选多样**而非单一路径。

### 2.3 安全提炼引擎（Safe Distiller · pi-xanthil 数据红线，文档 Phase 3）❗
pi-xanthil 特有约束（AGENTS.md 数据安全）：蒸馏**绝不碰 `draw_data` 原始行**。
- **输入仅限**：用户探索动作序列的「**查询骨架 Query Skeleton**」（SQL 结构，不含 WHERE 敏感值）+ API 调用拓扑 + 最终 report/business_requirements。
- **产物**：「💡 发现新提炼策略」子技能**提案**，在 RulesPane 由用户审核入库（Human-in-the-loop），杜绝数据泄漏污染。

## 3. 落到 pi-xanthil（三阶段，对位文档）

| 阶段 | 改造 | 域 |
|---|---|---|
| **运行时注入** | `MultiAgentExecutionPane` 引入 Planner 节点：解析需求出 Todo List，每步只「解锁」当前最需要的 2-3 个 skill 注入（执行效用排序+自适应预算+set-aware 渲染） | E |
| **子技能蒸馏** | skill-distillation.ts 扩：从成功轨迹切 2-5 动作子技能 + Trace2Skill 合并冗余 + SkillX 去重多样性 + 关系图 | E |
| **安全提炼** | Safe Distiller 后台任务（输入 query skeleton/API 拓扑/report，零 draw_data）→ RulesPane 提案确认 UI | D（红线域）+ E |

## 4. 与现有模块的边界

- **运行时注入 vs 现有 skill-retrieval**：现有多为语义检索；本条补「执行效用打分 + 自适应预算 + 瘦 context 渲染」，是升级不是另起。
- **子技能蒸馏 vs SkillOpt/memory→skill**：SkillOpt 优化**单个** skill 文档；本条做**多 skill 的切片/合并/去重/多样性**与关系图，互补。
- **Safe Distiller 触红线**：输入必须是脱敏骨架，落在 D 数据域，改完跑数据探索 LLM 隔离 grep。
- **零残留**：入池前产品无痕迹。

## 5. 将来开发要点（捞出指引）

1. **Quick Win = 运行时注入**：文档明确「核心发力点」。先在 MultiAgentExecutionPane 把「全量注入」改「每步 2-3 个 micro-skill 动态注入」，哪怕先用语义检索 + 固定预算，验证瘦 context 收益。
2. **执行效用信号**：注入排序从语义相似度升级为执行效用 Δ(t,s)——可复用 [[反馈效率度量 (EFC)]] 的反馈质量信号作为 Δ 的代理（接缝）。
3. **选技能看 full body**：SkillRouter 警示，路由别只喂标题。
4. **Safe Distiller 红线优先**：先把「只吃 query skeleton、零 draw_data」的输入边界做死 + 人审入库，再谈自动蒸馏，杜绝数据泄漏。
5. **去重优先多样**：子技能去重后优先选多样路径，防单一路径依赖。
