# 专题模块 操作手册

> **模块定位**：「专题」是 pi-Xanthil 面向**一个严肃业务问题**的深度工作台。它把「自由对话探索」与「BADIR + 门禁的流水线分析」做成**双模并列、互相 seed**，并复用日常模块的完整数据/报告链路——用于需要交付决策、经得起复核的专项分析（而非日常模块那种一次性的轻量探索）。
>
> 📅 更新日期：2026-06-20

---

## 0. 总览

进入路径：顶部主导航选中「**专题**」（`activeTab === "zhuanti"`），新建/进入一个专题任务（flow + session）。

专题与日常（探索）模块的关系：

| 维度 | 日常模块 | 专题模块 |
|---|---|---|
| 任务定位 | 一次性轻量探索、随手分析 | 围绕单一业务问题的深度专项 |
| 任务模型 | session（文档文件夹） | flow + session（flow 产物文件夹） |
| 核心交互 | 单一对话工作台 | **对话探索 ↔ 流水线** 双模互相 seed |
| 质量保障 | 自由，靠人把关 | 流水线**门禁**强制拦截不合格中间结果 |
| 沉淀 | 报告 / 记忆 | 报告 + **假设库飞轮** + **变更管理闭环** |

> 📌 专题与日常**互不串窜**：两类任务各自独立的 session / flow，侧栏分组管理，不会互相污染上下文。

界面布局分两层：

- **左侧竖栏（核心 5 项）**：对话探索 / 流水线 / 假设库 / 变更管理 / readme —— 专题独有的工作主线。
- **顶部横条（数据 / 报告链路）**：业务需求 / 原始数据 / 数据提取 / 聚合计算 / 聚合数据 / 数据探索 / 报告输出 / 汇报版本 / 报告审核 / 黄金策 / 行动 / DLF —— 与日常模块**同款 pane**，scope 切到当前专题 flow。

全部子 tab 一句话定位：

| 分区 | 子 tab | 一句话定位 | 与 LLM 关系 |
|---|---|---|---|
| 核心 | **对话探索** | 自由对话式探索问题，确认后一键 seed 流水线 | 主对话区，全 LLM |
| 核心 | **流水线** | AnaX：BADIR + 门禁的 9 节点线性分析流水线 | 各阶段 LLM + 确定性门禁判定 |
| 核心 | **假设库** | 沉淀被证实/证伪的假设（飞轮，下次复用） | 结构化存储，注入先验 |
| 核心 | **变更管理** | 决策建议转变更提案，跟踪落地与上游过时重跑 | 结构化跟踪 |
| 数据 | **业务需求** | 把模糊业务问题结构化为可分析需求文档 | LLM 抽取 |
| 数据 | **原始数据** 🔴 | 登记 `draw_data` 路径（红线域） | 原始行**禁入 LLM** |
| 数据 | **数据提取** | 用注册的提取工具把原始数据加工为聚合产物 | 工具只回灌聚合结果 |
| 数据 | **聚合计算** | 浏览器内本地聚合（无 LLM 通道） | **无 LLM**，可选生成 Python |
| 数据 | **聚合数据** 🟡 | 登记 `clean_data` 路径，左栏只读注入 | 允许进 LLM |
| 数据 | **数据探索** 🔒 | DuckDB-WASM 拖拽 BI | **永久禁止 LLM** |
| 报告 | **报告输出** | 登记 / 浏览 `report` 产物目录 | 衍生产物 |
| 报告 | **汇报版本** | 详细报告精炼为汇报版 + 故事线 | LLM 改写 |
| 报告 | **报告审核** | 多模型打分 + 修改建议 + 自动修复 | LLM 评审 |
| 报告 | **黄金策** | 商业框架（决策树/TOC/SWOT…）生成策略图 | 多模型并行 |
| 报告 | **行动** | 报告建议拆为可执行 action item | LLM + 任务管理 |

> ⚠️ 数据安全分级见 `AGENTS.md §一`。「数据探索」tab 永久禁止任何 LLM 调用，**这是项目的核心安全契约**，专题模块同样严格遵守。

## 1. 对话探索（`anax_chat`）

### 功能

- 专题专属的**独立对话 session**（与日常模块对话互不串窜），用于在正式跑流水线之前，**自由地把问题聊清楚**：澄清诉求、碰撞初始假设、确认口径。
- **左侧**：`CleanDataDocsColumn`（当前专题 flow 的 `clean_data` 只读文档竖栏，可折叠复制），与日常模块「数据分析」工作视图同款范式。
- 顶部按钮「**启动假设验证流水线**」：在对话沉淀出明确问题与初始假设后，点击 → 系统据对话自动起草一份**流水线 seed**（问题 + 初始假设）。
- seed 弹窗可**人工编辑**后点「填入流水线」→ 自动跳到「流水线」tab 并把 seed 填进 brief。⚠️ **不会自动运行**，给你最后一次确认的机会。

### 典型操作

1. 在对话里把业务诉求聊透（如"华南区会员 6 月留存率环比 -2%，到底是拉新质量、渠道流失还是季节性"）。
2. 确认 3 个左右可证伪的初始假设。
3. 点「启动假设验证流水线」→ 审视/微调自动起草的 seed → 点「填入流水线」。
4. 跳到「流水线」tab，确认 brief 后再手动运行。

### 双模回流

流水线跑完后，其摘要会**自动回流**到对话探索（`pushZhuantiChatSummary`），你可以基于流水线结论继续追问、发起下一轮探索——形成「对话 → 流水线 → 对话」的螺旋。

## 2. 流水线（`anax_view`，AnaX 商业分析方法论）

> 流水线是专题模块的**分析主引擎**。它把一次严肃的商业数据分析拆成 **8 个阶段（BADIR + 执行验证 + 归档，共 9 个节点）**，并在关键节点设**质量门禁**——不合格的中间结果会被门禁**拦下、流程在该处停止**，而不是带着问题一路滑到最终建议。

### 前提：先登记聚合数据

AnaX 只读「聚合数据」里已登记的 `clean_data` 文件，**不碰原始明细**。若当前专题 flow 里没有与问题相关的聚合数据，流程会在**数据质量门禁**被红灯拦下——这是最常见的卡点，不是 bug，而是逼你先把数据备齐（用「数据提取 / 聚合计算」把 `draw_data` 加工成 `clean_data`）。

### 9 个节点

| 步 | 节点 | 角色 | 类型 | 产物 | 做什么 |
|---|---|---|---|---|---|
| B | 商务问题 | 协调官 | agent | `01-brief.md` | 把模糊诉求结构化为「时间+对象+指标+变化幅度+目的」，提出 ≥3 个可证伪假设 |
| A | 分析规范 | 规范官 | agent | `02-spec.md` | 选匹配的统计方法，假设写成可验证形式，列聚合数据需求，预声明局限性 |
| D | 数据质量 | 数据策展官 | agent | `03-data-quality.md` | 对已登记聚合数据做 6 维度评分（完整/准确/时效/一致/有效/唯一） |
| 🚦 | **数据质量门禁** | 复核官 | gate | — | 检查综合分、局限披露、置信度与证据是否达标，不达标红灯停车 |
| I | 假设验证与洞察 | 假设分析师 | agent | `04-insights.md` | 逐一检验假设，报告 p 值+效应量+置信区间，给 ✅/❌/⚠️ 结论，高影响假设交叉验证 |
| R | 决策建议 | 战略官 | agent | `05-recommendations.md` | 洞察转建议包，每条含 7 要素（做什么/为什么/负责人/时间/成功标准/验证方案/收益与风险） |
| 🚦 | **两阶段复核门禁** | 复核官 | gate | — | 方法论审查 + 业务审查，两轴都过才放行 |
| X | 执行验证 | 验证官 | agent | `08-verify.md` | 建议落地后回收实际效果，对比成功标准，回算 ROI |
| 📦 | 归档与沉淀 | 协调官 | agent | `09-archive-summary.md` | 归档结论，沉淀证实/证伪的假设到假设库，标注业务场景供复用 |

### 门禁是怎么工作的

门禁节点让 AI 对上游产出做结构化审查，系统再按**固定阈值确定性判定**（AI 自己说"通过"不算数）：

- 置信度需 ≥ **medium**
- 每个结论的证据数需 ≥ **2**
- 数据质量综合分需 ≥ **7**（满分 10）

只要命中任意一条**质量红线**或阈值不达标，门禁亮 🔴 BLOCKED，流程在该节点停下；按提示补数据/修内容后，从该节点重跑（继承上游产物，不必从头来）。

### 7 条质量红线

| 编号 | 描述 |
|---|---|
| RL01 | 结论无数据证据支撑 |
| RL02 | 假设未经验证直接采纳 |
| RL03 | 数据质量综合评分 < 5 仍继续 |
| RL04 | 建议未回应原始商务问题 |
| RL05 | 关键局限性未披露 |
| RL06 | 高影响假设（>¥500K/月）未交叉验证 |
| RL07 | 建议缺少负责人 / 时间 / 成功标准 / 验证方案 |

### 操作步骤

1. **seed 来源**：从「对话探索」填入，或在 brief 直接写清商务诉求。
2. **实例化**：首次运行把方法论物化成一条真实工作流（9 节点线性 DAG）。完整版 = AnaX v3.0；想快可选 Quick 版（保留核心阶段与门禁，省去部分交叉验证/归档）。
3. **运行**：9 阶段实时推进，每个门禁亮 🟢 通过 / 🔴 阻断（被阻断会显示具体原因：哪条红线、哪个阈值）。
4. **修正重跑**：门禁红灯时按提示补数据或修内容，从该节点重跑。
5. **导出报告**：全流程通过后，各阶段交付物拼成报告导出；证实/证伪的假设自动沉淀进**假设库**。

### 案例走一遍：会员留存率下滑 2%

| 阶段 | 走法 | 裁决 |
|---|---|---|
| B 商务问题 | 结构化为：对象=华南区会员、指标=月留存率、变化=环比 -2pct；提 3 个可证伪假设（拉新质量↓/某渠道流失/季节性） | — |
| A 分析规范 | 选同期群(cohort)留存分析 + 渠道分组对比；声明局限：只有聚合数据、无个体行为日志 | — |
| D 数据质量 + 门禁 | 对留存聚合表打 6 维度分。若没登记留存相关聚合数据 → 评分过低 → 🔴 数据门禁阻断 | 🔴 阻断 |
| I 假设验证 | 数据达标后：渠道 A 留存显著低于均值（p<0.05、效应量中等）→ 假设2 ✅，假设1/3 ❌ 证据不足 | — |
| R 决策建议 + 复核门禁 | 产出建议包（收紧渠道 A 投放 + 针对性召回，含四要素）→ 复核通过 | 🟢 通过 |
| X 执行验证 + 归档 | 落地后回收效果、回算 ROI；归档"渠道 A 拉新质量→留存"因果链供复用 | — |

> 提示：门禁是为了挡住"看起来对、其实没站稳"的结论。被拦下不是出错，而是流程在保护你的决策质量。

## 3. 数据 / 报告链路（顶部横条，复用日常模块）

这一段 tab 与日常（探索）模块**同款 pane**，仅 scope 切到当前专题 flow。详细功能/操作/案例见**日常模块 readme**，此处只列专题语境下的一句话用法：

- **业务需求**：把专题问题结构化为需求文档，`dataNeeds.fields` 可单向跳「数据探索」做字段验证。
- **原始数据** 🔴：登记 `draw_data` 路径（仅登记不复制）；原始行级数据**禁入 LLM**。
- **数据提取**（`extraction`）：用注册的提取工具（`/api/extraction-tools/:id/run`）把原始数据加工为聚合产物，工具**只回灌聚合结果**，原始明细不进 LLM。
- **聚合计算**（`aggregate_compute`）：浏览器内本地聚合（sum/avg/min/max/count、时间粒度、分组），**无 LLM 通道**；可选生成等价 Python 脚本。是补足"数据备齐"的本地快通道。
- **聚合数据** 🟡：登记 `clean_data` 路径，允许进 LLM；在「对话探索」左栏只读展示。
- **数据探索** 🔒：DuckDB-WASM 浏览器内拖拽 BI + 字段剖析 + JOIN，**永久禁止 LLM**。
- **报告输出 / 汇报版本 / 报告审核 / 黄金策 / 行动**：与日常模块一致的报告生产、精炼、评审、策略图、行动拆解链路，产物落当前专题 flow 目录。

> 📌 校验数据探索子树无 LLM 调用（专题同样适用）：
>
> ```bash
> grep -rE "(generate|chat|extract|clarify|sink|distill).*api\." \
>   web/src/components/DataExplorationPane.tsx \
>   web/src/components/data-exploration/
> # 应无任何匹配
> ```

## 4. 假设库（`hypothesis`，飞轮）

### 功能

- 按**业务场景（scene）分组**沉淀假设条目，每条含：假设陈述、裁决（✅ 成立 / ❌ 不成立 / ⚠️ 部分）、证据、影响。
- 流水线归档阶段会把被证实/证伪的假设**自动写入**假设库；也可手动新增。
- 下次分析同类问题时，假设库作为**先验注入**——证实过的因果链直接复用，不必从零再猜，越用越快（这就是"飞轮"）。

### 典型操作

1. 流水线跑完后，到假设库查看本次自动沉淀的条目。
2. 对长期复用的关键假设，手动补充证据/影响，归到清晰的 scene。
3. 下个专题遇到同 scene 问题时，已有假设作为起点。

## 5. 变更管理（`change_mgmt`，闭环）

### 功能

- 把决策建议转成**变更提案**进入跟踪，状态流：待审批 `proposed` → 已批准 `approved` → 已落地 `applied`（或 `rejected` 驳回）。
- 提案可**关联 AnaX 节点**（来源 `AnaX v3.0`），记录每条建议从哪个分析阶段而来。
- **过时检测（staleNodes）**：当上游节点变更，依赖它的下游节点会被标记为"过时需重跑"，避免拿着旧结论继续往下走。
- 形成「分析 → 建议 → 执行 → 验证」的闭环，而不是一份读完即弃的报告。

### 典型操作

1. 流水线产出建议后，挑可落地的转为变更提案。
2. 走审批：待审批 → 批准 → 落地（落地时填实际结果）。
3. 若上游数据/结论有变，看 staleNodes 提示，定位需重跑的下游节点。

## 双模端到端最佳实践流（会员留存下滑专题）

```
1. 对话探索        → 自由聊清问题 + 碰 3 个初始假设 → 点「启动假设验证流水线」起草 seed
2. 业务需求        → （可选）把诉求结构化为需求文档，拿到 dataNeeds.fields
3. 原始数据        → 登记留存明细 csv（不进 LLM）
4. 数据提取/聚合计算 → 把明细加工成留存聚合表（本地通道，落 clean_data）
5. 聚合数据        → 登记聚合表，左栏可见
6. 流水线（AnaX）  → 确认 seed brief → 运行 9 节点 → 数据门禁 🔴 卡点先补数据 → 重跑 → 复核 🟢 通过
7. 报告链路        → 报告输出 → 报告审核自动修复 → 汇报版本出故事线 → 黄金策给策略图 → 行动拆 action
8. 假设库          → 归档自动沉淀「渠道 A 拉新质量→留存」因果链
9. 变更管理        → 把"收紧渠道 A 投放"转变更提案，跟踪落地与验证
10. 对话探索        → 流水线摘要回流，基于结论继续追问下一轮
```

每一步产物都是下一步输入，所有 LLM 产物落当前专题 flow 目录自动归档，不污染 `draw_data`，不破坏数据安全分级。

| 操作 | 允许吗？ |

| 在「数据探索」加载 csv 用于剖析 | ✅ 浏览器内，无 LLM |

| 「聚合计算」本地跑聚合 | ✅ 无 LLM 通道 |

## 故障排查（FAQ）

- **流水线卡在数据质量门禁**：当前专题 flow 没登记相关聚合数据，或综合分 < 7。先用「数据提取/聚合计算」备齐 `clean_data` 再从该节点重跑。
- **对话探索点不动「启动流水线」**：需要先有对话内容（消息数 > 0）才能起草 seed。
- **流水线复核门禁红灯**：结论缺证据（每条 < 2 条）或建议四要素不全；补齐后重跑该节点。
- **假设库没自动沉淀**：流水线未走到归档阶段（被门禁拦在中途），或被 Quick 版省略；通过完整版跑完即沉淀。
- **变更提案标了过时（stale）**：上游节点有新变更，按 staleNodes 提示重跑对应下游。
- **token 撑爆**：对话探索点「压缩上下文」`compactZhuantiChatContext()`。
