> **状态**：方案待执行 · 立项 2026-06-13 · 总控(Claude)持有，Agent-E 实现
> **目标**：借"植入 agent-loop 能力"这次迭代，把工作流模块从「三套重叠执行栈 + 大量孤儿代码」收敛为「单引擎闭环系统」。

## 1. 改造目标与原则

1. **单引擎**：所有 flow 执行统一走 `runMultiAgent`，单 agent = "1 节点的 DAG"，不再有第二条执行实现。
2. **零孤儿**：清掉 ③，活跃代码只面对一条真路径。
3. **闭环就位**：在收口后的单引擎上实现 `onBlock` 反馈回边 + 预算，loop 能力一次性惠及主栈 + AnaX + 未来所有 flow。
4. **顺序不可逆**：先 0、再 1、再 2。跳过 0/1 直接做 2 = 在分叉双栈 + 巨型单体上叠 loop，债务加倍。
5. **最小改动 + 向后兼容**：现有 multi flow 与 AnaX flow 行为不回退；`Flow.kind` 列保留（存量全为 multi），只停止产出 single。

## 2. 阶段 0 · 清场（纯减法，低风险）

### 删除清单（删前再跑一次全量引用 grep 复核，遵删除确认铁律）
- 前端 5 文件：`ExecutionPane` / `AgentFlowPane` / `FlowChatPane` / `FlowWorkflowPane` / `FlowEditorPane`
- 后端：`handleExecuteFlow` + `index.ts:5458` 的 `execute_flow` dispatch 分支 + `types.ts` 的 `execute_flow` ClientMessage 成员
- single-kind 退化分支：`CreationPane` 内 `p.kind === "multi" ? … : …` 的 single 文案分支（组件简化为只服务 multi）；`newFlow`/`onNewFlow`/`CreationPane` 的 `kind` 入参收敛为常量

### 完成标准
- `npm run typecheck` + `npm run build` 全绿
- `multi` tab 与 AnaX tab 手动冒烟无回归（新建 flow → 编辑 DAG → 跑一次 → run 历史 → 恢复）
- 收益：-~1800 行前端 + -~110 行后端，搜索/认知噪声骤降

## 3. 阶段 1 · 收口（双栈并一引擎 + 路由归位 + UI 去重）

### 3.1 执行路径统一
- 废弃 inline 单 agent 路径；若仍需"无 workflow.json 时单轮执行"的退路，改为**在 `runMultiAgent` 入口处合成一个 1 节点 workflow**，而非走独立 handler。
- 结果：gate / fan-out / specs / resume / 未来的 loop **对所有执行一致可用**。

### 3.2 路由从 legacy index.ts 抽到 routes/engine.ts
- 现状：22 条 flow 路由 + 2 个 WS handler 压在 `index.ts`，`routes/engine.ts` 是空壳（因"需访问 activeRuns/wss"被暂留）。
- **前置依赖（接缝层，归总控）**：先抽 `server/src/runtime.ts`，暴露 `activeFlowRuns`/`activeMultiAgentRuns`/`wss`/`send` 等运行时句柄；之后 flow 路由与 WS handler 才能迁出 index.ts。
- 迁移只搬运不改逻辑，typecheck/build/现有测试须绿。

### 3.3 前端执行订阅/恢复逻辑（AnaX 前端**不去重**，仅工作流主栈内部清理）
> **决策（2026-06-13）**：AnaX 定位为「可能演化成独立产品 / 后台系统」（白皮书 type-B：后台常驻/定时/多 run 决策编排，运行语义会大幅分叉）。因此**不把 AnaX 前端与工作流模块耦合**。
- **缓做** `useFlowRun(flowId)` 跨 pane 共享 hook。`MultiAgentExecutionPane` 与 `AnaXPane` 的 WS 订阅/恢复重复（~150 行）**暂留**——这点重复是 AnaX 独立演化的「便宜保险」，强行抽共享 hook 会过早锁死接缝。
- 仅在**工作流主栈内部**做收敛：把 `MultiAgentExecutionPane`(1296 巨型单体) 按职责拆（DAG 编辑 / 节点编辑器 / 执行面板 / run 历史 / tool 配置），把"执行+恢复"剥成**主栈私有** hook，编辑器与执行视图解耦。AnaX 不参与。
- 若将来 AnaX 轨迹明确且运行语义未分叉，再回头评估是否抽**薄传输底座** `useFlowRunStream`（纯 WS+状态机、零 UI 意见，挂载点无关）供两者各自 opt-in。

### AnaX 与本次改造的边界（重要）
- **引擎层共享保留且强化**：AnaX 继续走 `runMultiAgent`，阶段 2 的 `onBlock` 闭环**白嫖到手**——这正是「智能决策闭环系统」需要的 verifier+反馈+预算。引擎共享是 AnaX 的**战略支撑**，不是耦合负担。
- **产品层 AnaX 独占、本次完全不碰**：9 阶段进度 / 🔴🟢 门禁卡 / 假设飞轮 / change proposals / 阈值 / BADIR+X 方法论。
- **挂载点解耦**：AnaX 将来从实验室提升为独立顶级模块、乃至独立产品/后台系统时，仍消费共享引擎；其前端不依赖工作流模块，迁移无牵连。
- **type-B 后台/调度**（AnaX 未来方向）**不在本次 loop 范围**；届时引擎若需支持后台常驻/多 run 编排，应让 `runMultiAgent` 暴露原语，另案设计。

### 完成标准
- 单引擎：grep 确认无第二处 topoOrder 执行实现
- `routes/engine.ts` 承载 flow 路由，index.ts 不再有 flow 业务路由（流式 handler 经 runtime.ts 解耦）
- 工作流主栈执行/恢复逻辑收敛进主栈私有 hook；**AnaXPane 不改、不与主栈共享前端逻辑**
- 全绿 + 双栈冒烟无回归（AnaX 行为零变化）

## 4. 阶段 2 · 加闭环（落 agent-loop backlog 方案）

> **权威口径**：`docs/工作流-onblock契约.md`（T-C3 已交付）。以下为概要，实现以契约文档为准。

在收口后的**单引擎**上实现 `docs/backlog/agent-loop-工作流闭环.md` §3 草案：

- **类型**：`WorkflowNode.onBlock { retryFromNodeId, maxIterations?, feedbackVar? }`（仅 `kind:"gate"` 生效；不配置=现行 halt-or-pass，向后兼容）。
- **runner**：`runMultiAgent` 主循环（`multi-agent-runner.ts:364`）从顺序遍历改为带"回跳指针"的索引循环；gate blocked 且预算未尽时，把 `verdict.reasons` 写入 `blackboard[feedbackVar]`、重置回跳区间的脏产物、游标拨回 `retryFromNodeId`；预算耗尽才真正中断升级人工。
- **预算/停止条件**：`maxIterations` + 复用 `cache.ts` 的 token/cost 累计做 run 级阈值 + 红线命中硬中断不进重试环 + 人工续跑（沿用 resume）。
- **trace**：每轮迭代产物落 `runDir/<nodeId>/iter-<n>/`，verdict 落 `gates/<id>-iter<n>.json`，保留迭代谱系。
- **前端**：`WorkflowDagEditor` 节点编辑器给 gate 节点加 retry 目标 + 预算配置；执行面板（经 `useFlowRun`）展示迭代轮次。**不需要新 tab、不需要新模块。**
- **MVP**：SQL 修复 loop（`plan→sql→run_sql(tool)→sql_gate(onBlock 回 sql)`）。

### 完成标准
- `multi-agent-runner.test.ts` 新增确定性单测：blocked→回跳→第二轮 pass；预算耗尽→中断（用既有 fake adapter 注入，无需真 pi）
- MVP SQL loop 用真实可执行 SQL 工具节点 + 真实聚合数据**实跑收敛**（吸取 AnaX "fake 跑通≠E2E" 教训）
- 现有 multi flow / AnaX flow 不配 onBlock 时行为零变化

## 6. 执行顺序与风险

| 阶段 | 风险 | 把控 |
|---|---|---|
| 0 清场 | 误删活跃代码 | 删前全量 grep 复核 + 全绿 + 双栈冒烟；`Flow.kind` 列不动免数据风险 |
| 1 收口 | runtime.ts 抽取触接缝、流式句柄解耦易出错 | 接缝层归总控、只搬不改、每批 typecheck/build/test 绿 |
| 2 加闭环 | 回跳逻辑死循环 / 脏产物未清 | `maxIterations` 硬上限 + 回跳区间产物重置 + 确定性单测先行 |

**总原则**：每阶段独立可交付、可回滚；阶段间以 typecheck+build+冒烟绿为门禁，吸取 P0-C "fake 跑通≠E2E 验证" 教训，loop MVP 必须真实实跑收敛。
